Bubble Bath Auth System

Go Project Structure

bubble-bath/
├── cmd/server/main.go          # Entry point
├── internal/
│   ├── auth/                   # Login, signup, session logic
│   ├── crypto/                 # AES-256-GCM token encode/decode
│   ├── hsv/                    # Distance calc, tolerance algorithm
│   ├── handlers/               # HTTP route handlers
│   ├── models/                 # User, Token, Profile structs
│   ├── store/                  # PostgreSQL queries
│   └── recovery/               # Rotating code validator
├── web/                       # React frontend (separate module)
│   ├── src/components/      # ColorPicker, NumberInput, LoginForm
│   ├── src/three/           # Token visualization (future)
│   └── src/pages/           # Signup, Login, Profile
├── migrations/               # PostgreSQL schema migrations
├── config/                   # Environment config loading
├── go.mod
├── go.sum
└── Dockerfile

Key Design Notes

internal/ package

Go convention — code in internal/ cannot be imported by external projects. Your crypto and tolerance logic stays private.

Redis for sessions

Refresh tokens and rate limiting live in Redis. Fast, TTL-native, no need to garbage-collect expired sessions in Postgres.

Separate frontend

React app builds to static files, served by Go or a CDN. Clean separation — Go never renders HTML.

Future: Google Cloud

Go binary → Cloud Run. Postgres → Cloud SQL. Redis → Memorystore. Single Dockerfile, no rewrites needed.
